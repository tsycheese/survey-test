# 技术笔记：解决行内编辑（Inline Editing）中的高度跳变问题

在开发问卷编辑器的过程中，我们遇到了一个典型的 UI 交互问题：当用户点击题目标题从“预览态”（`div`）切换到“编辑态”（`textarea`）时，虽然两者设置了相同的 `height` 或 `min-height`，但父容器的高度会发生微小的抖动（从 36px 变为 42.5px）。

## 1. 现象描述
- **预览态**：使用 `div` 展示文本，高度稳定在 36px。
- **编辑态**：使用 `textarea` 进行编辑，设置 `min-h-[36px]`，但其父级 `flex-1` 元素却被撑开到了 42.5px。
- **结果**：切换时页面元素发生垂直位移，产生廉价的“闪烁感”。

## 2. 根源分析：`inline-block` 的幽灵间距
问题的核心在于 `textarea` 默认的布局行为：

1. **基线对齐（Baseline Alignment）**：
   `textarea` 默认被视为 `inline-block` 元素。在 CSS 的行内布局逻辑中，浏览器会将该元素的基线与行内文字基线对齐。为了给文字的“降部”（descender，如字母 g, j, y 的下半部分）留出空间，浏览器会在元素下方预留几个像素的“幽灵间距”。

2. **父容器撑开**：
   即便 `textarea` 本身是 36px，父容器（`flex` 或 `block`）为了容纳这个带有基线偏移的行内元素，会被迫增加总高度，导致了 42.5px 的出现。

## 3. 解决方案（最佳实践）

### 方案 A：强制块级化（推荐）
最直接、最彻底的办法是将 `textarea` 从行内逻辑中抽离：
```css
/* 在 Tailwind 中使用 block 类名 */
textarea {
  display: block;
  width: 100%;
}
```
**原理**：一旦变为 `block` 元素，它就不再遵循基线对齐规则，父容器会严丝合缝地包裹住它的物理高度。

### 方案 B：统一行高与 Padding
确保预览态和编辑态的文字渲染环境完全一致：
- **固定行高**：使用绝对值（如 `leading-6` 即 24px）而非相对值。
- **边框补偿**：`textarea` 的 `scrollHeight` 不包含 `border`。在动态计算高度时，需注意 `box-sizing: border-box` 下的边框占用。
- **最小高度锁定**：为两者设置完全一致的 `min-height`。

## 4. 经验总结
在实现“所见即所得”的行内编辑器时：
1. **永远检查 `display` 属性**：默认的 `inline-block` 是大多数对齐问题的根源。
2. **视觉镜像化**：编辑态和预览态不仅要长得像，**布局模型（Layout Model）也必须一致**。
3. **善用开发者工具**：当父元素高度溢出时，检查是否由于 `textarea` 的 `baseline` 导致了多余的 `bottom` 间距。
